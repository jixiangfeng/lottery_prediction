# 彩票预测系统三层设计文档

## 1. 背景

当前 `projects/lottery_prediction` 项目已经形成了一套可复现的三位彩研究系统，核心是 `fc3d` / `pl3` 的固定评分、参数搜索、冻结测试和日报生成流程。现有实现的优点是边界清晰、指纹完备、评估隔离明确，但它的表达能力仍然偏单一：主要围绕一个 learned ranker v4 做历史拟合、参数调优和固定前推。

如果目标不是“做一个看起来很复杂的模型”，而是“建立一个更稳、更可比、更少自欺的研究体系”，那么更好的方向不是单纯加大模型复杂度，而是把系统拆成三层：

1. 基线层：回答“到底有没有信号”
2. 研究层：回答“哪类方法稳定优于基线”
3. 实战层：回答“哪些结果可以冻结并长期执行”

这个文档给出一版可落地的三层设计。

## 2. 设计目标

### 2.1 核心目标

- 建立统一的数据入口和特征口径。
- 为所有模型提供一致的时间切分、回测协议和评估指标。
- 把“预测好不好”拆成多个可量化维度，而不是只看 TopK 命中。
- 避免未来数据泄漏，保证评估可信。
- 允许低成本增加新模型，同时不破坏现有的可复现性。

### 2.2 约束目标

- 训练、验证、测试必须严格按时间切分。
- Frozen Test 只能做最终评估，不能参与选参。
- 所有参数、特征配置、数据版本、源码版本都要可指纹化。
- 所有输出都要能回溯到数据、参数和代码。
- 默认仅支持 `fc3d`、`pl3`，不把 `pl5` 当作这条主线的目标玩法。

### 2.3 非目标

- 不承诺中奖。
- 不承诺盈利。
- 不追求模型越复杂越好。
- 不把单次测试的偶然结果当作结论。

## 3. 总体架构

### 3.1 分层视图

系统分为三层：

#### A. 基线层
用于建立最朴素的对照组，检测历史中是否真的存在可利用信号。

#### B. 研究层
用于比较不同模型、特征、窗口和聚合方式，产出可复现的实验结果。

#### C. 实战层
用于冻结少量长期稳定的策略，形成固定产物和固定输出流程。

### 3.2 数据流概览

```text
原始抓取 JSONL / 本地标准化 CSV
  -> 标准化与校验
  -> 时间切分
  -> 基线层 / 研究层 / 实战层
  -> 统一评估
  -> 参数/报告/指纹产物
```

### 3.3 推荐策略

- 基线层先跑通。
- 研究层再做模型竞争。
- 实战层只接收长期稳定胜出的方案。

## 4. 数据层设计

### 4.1 输入数据

统一使用本地 CSV，字段标准化为：

- 三位彩：`期数, 百位, 十位, 个位`
- 五位彩：`期数, 万位, 千位, 百位, 十位, 个位`

当前主线只围绕三位彩展开。

### 4.2 数据清洗规则

必须校验：

- 期号是否为纯数字。
- 是否存在重复期号。
- 每期开奖结果是否满足玩法位数要求。
- 每一位号码是否在合法范围内。
- 数据是否按数值期号可稳定排序。

### 4.3 标准化输出

统一生成：

- 标准列名。
- 期号字符串化后的固定格式。
- 按期号升序/降序排序结果。
- canonical 数据指纹。

### 4.4 数据指纹

建议保留三类指纹：

- `csv_sha256`：原始 CSV 文件级指纹。
- `canonical_data_sha256`：标准化语义数据指纹。
- `source_fingerprint`：源码与关键实现指纹。

意义是：即使文件名相同，只要数据内容、规则或源码变了，结果就必须被视为不同实验。

### 4.5 当前阶段存储原则

当前阶段不引入 MySQL 作为主存储。原因是系统还处在研究和真实预测前验证阶段，过早数据库化会增加迁移、Schema 维护和查询层复杂度，反而不利于快速验证模型是否真的有效。

推荐采用文件系统优先：

- 原始抓取结果保存为 JSONL。
- 标准化历史数据保存为 CSV。
- 参数、评估、日报、搜索结果保存为 JSON / Markdown。
- 大型中间产物、模型文件、缓存继续放文件系统。

### 4.6 原始抓取数据使用 JSONL

原始爬取的开奖号建议使用 JSONL 保存，每行一条抓取到的原始记录。

JSONL 比直接覆盖 CSV 更适合作为原始层，原因是：

- 支持逐行追加，便于增量抓取。
- 每条记录能保留完整来源字段和原始响应片段。
- 适合做审计和问题回放。
- 单行损坏不会影响整个文件读取。
- 易于用脚本转成标准化 CSV。

推荐目录：

```text
data/
  fc3d/
    raw/
      cwl_history.jsonl
    official_history.csv
  pl3/
    raw/
      sporttery_history.jsonl
    official_history.csv
```

### 4.7 JSONL 推荐字段

每一行建议包含：

```json
{"lotteryCode":"fc3d","issue":"2026188","drawDate":"2026-07-18","digits":[1,2,3],"sourceName":"cwl.gov.cn","sourceUrl":"https://...","fetchedAt":"2026-07-19T09:00:00+08:00","raw":{"...":"..."},"rawSha256":"..."}
```

字段说明：

- `lotteryCode`：彩种，例如 `fc3d`、`pl3`。
- `issue`：期号，统一字符串。
- `drawDate`：开奖日期。
- `digits`：标准化后的号码数组。
- `sourceName`：来源名称。
- `sourceUrl`：来源 URL。
- `fetchedAt`：抓取时间。
- `raw`：原始响应中的单条记录。
- `rawSha256`：原始记录语义指纹。

### 4.8 从 JSONL 到 CSV 的转换

数据层建议拆成两段：

```text
抓取器
  -> raw/*.jsonl
  -> reconcile/normalize
  -> official_history.csv
  -> canonical data fingerprint
```

训练、评估和日报仍然只读取标准化 CSV，不直接读取网络，也不直接依赖 JSONL。JSONL 是原始证据层，CSV 是研究计算层。

### 4.9 数据更新与对账流程

数据更新必须先保证正确性，再考虑自动化。推荐流程如下：

1. 抓取阶段

   - 只允许固定白名单来源访问网络。
   - 每次抓取追加写入 raw JSONL，不直接覆盖标准化 CSV。
   - 每条 raw 记录必须包含彩种、期号、开奖号、来源、抓取时间、原始记录指纹。

2. 去重阶段

   - 以 `lotteryCode + issue + sourceName` 作为原始记录去重键。
   - 同一来源同一期多次抓取内容一致时允许合并。
   - 同一来源同一期内容不一致时标记为冲突，不自动覆盖。

3. 对账阶段

   - 同一期多来源开奖号一致，才允许自动进入标准化候选集。
   - 多来源开奖号不一致时，必须进入人工确认清单。
   - 单来源数据可以进入候选集，但报告中应标记来源数量和可信级别。

4. CSV 生成阶段

   - 只从已确认的标准化候选集生成 `official_history.csv`。
   - 生成前必须校验期号、位数、数字范围、重复期号、排序稳定性。
   - 生成后计算 canonical data fingerprint。

5. 追加阶段

   - 允许在当前 CSV 尾部追加新开奖。
   - 不允许静默修改历史期号或历史开奖号。
   - 如果必须修正历史数据，必须生成新的数据指纹，并使旧实验产物失效。

6. 人工确认触发条件

   - 同一期多来源结果不一致。
   - 同一来源同一期多次抓取结果不一致。
   - 期号缺口异常。
   - 开奖日期和期号顺序不一致。
   - CSV canonical 指纹与预期不一致。

### 4.10 MySQL 延后引入

MySQL 暂时移出主设计。等满足以下条件后再考虑引入：

- 真实预测流程已经跑通。
- 已经有稳定的每日预测和开奖后追踪。
- 需要后台页面、长期统计、策略版本检索或多人查询。
- 文件系统产物已经多到不方便人工管理。

届时 MySQL 只作为结构化事实库和查询索引，不替代 JSONL、CSV、报告和模型产物文件。

## 5. 基线层设计

### 5.1 目标

基线层不追求强预测，只回答一个问题：

**复杂模型到底有没有比朴素规则更好。**

如果连基线都打不过，后续复杂方法没有讨论价值。

### 5.2 基线模型建议

#### 5.2.1 均匀随机基线

- 所有号码等概率。
- 用于定义理论下限和随机参考线。

#### 5.2.2 最近窗口频率基线

- 统计最近 `N` 期各位置号码频率。
- 用频率排序作为候选排序依据。

#### 5.2.3 长窗口频率基线

- 使用更长窗口观察长期分布。
- 用于和短窗口基线比较稳定性。

#### 5.2.4 位置独立基线

- 每一位单独建模。
- 假设百位、十位、个位近似独立。

#### 5.2.5 形态基线

- 和值。
- 跨度。
- 组三 / 组六 / 豹子。
- 奇偶比。
- 大小比。

#### 5.2.6 简单转移基线

- 统计前一期形态到下一期形态的转移关系。
- 统计位置分布变化。

### 5.3 基线输出

每个基线都应输出：

- 候选排序。
- 概率或打分。
- TopK 列表。
- 统一评估指标。

### 5.4 基线评估指标

建议至少保留：

- `logloss`
- `Brier score`
- `mean rank`
- `mean rank percentile`
- `TopK hit rate`
- `calibration` / 校准曲线
- 分时间块稳定性

### 5.5 基线层的价值

基线层的作用不是追求高分，而是防止研究层误判。

如果一个复杂模型只是在某个切分上“看起来更好”，但在基线层下没有持续优势，那它大概率只是噪声拟合。

## 6. 研究层设计

### 6.1 目标

研究层是“实验室”，用于比较方法，不直接决定最终实战输出。

### 6.2 研究对象

可在这一层比较：

- 线性 ranker
- LightGBM / XGBoost
- 小型序列模型
- 简单集成模型
- 不同特征窗口和衰减配置
- 不同组选聚合方式

### 6.3 统一实验协议

所有研究模型必须共享：

- 同一份标准化数据。
- 同一套时间切分。
- 同一套评估指标。
- 同一套随机种子管理。
- 同一套指纹机制。

### 6.4 推荐的实验切分

#### 6.4.1 Search

用于粗搜候选特征和参数。

#### 6.4.2 Validation

只对 Search 已经锁定的单组参数做确认，不得比较候选并反向选参。

#### 6.4.3 Frozen Test

用于一次性最终评估。

Frozen Test 不能参与：

- 特征增删
- 窗口选择
- 权重选择
- temperature 选择
- 候选预算选择
- 目标函数选择
- gate 选择

### 6.5 研究层推荐模型

#### 6.5.1 线性 ranker

适合做强解释对照组，优点是简单、稳定、容易做指纹。

#### 6.5.2 树模型

适合做非线性交互对照组，但要特别警惕过拟合。

#### 6.5.3 小型序列模型

适合验证时间依赖是否真的存在，但应当严格控制容量。

#### 6.5.4 集成模型

把多个弱模型加权，通常比单一复杂模型更稳。

### 6.6 研究层评价原则

研究层不是看“最高分”，而是看：

- 多个时间切分是否都稳定。
- 是否持续优于基线。
- 是否在不同指标上都更稳。
- 是否在 Frozen Test 上仍保持一致优势。

### 6.7 研究层输出

建议产出：

- 参数 JSON
- 搜索报告 JSON
- 验证报告 Markdown
- 冻结评估 JSON / Markdown
- 指纹文件
- 训练/评估日志

## 7. 实战层设计

### 7.1 目标

实战层只保留真正长期稳定的结果，避免频繁换模型、换参数、换解释口径。

### 7.2 实战层准入条件

一个策略进入实战层前，应至少满足：

- 多次 walk-forward 稳定优于基线。
- Frozen Test 表现没有明显回落。
- 指标不是单靠某一项偶然拉高。
- 在多个时间块里都能保持相对稳定。

### 7.3 实战层产物

实战层建议冻结以下内容：

- 固定特征配置
- 固定参数配置
- 固定输入数据版本
- 固定评估协议
- 固定输出格式
- 固定报告模板

### 7.4 实战层风险控制

- 限制策略数量。
- 限制人工频繁干预。
- 限制参数漂移。
- 限制对 Frozen Test 的反复查看。

### 7.5 实战层输出

- 每期固定预测结果。
- 每日/每周总结。
- 策略稳定性追踪。
- 版本变更记录。

### 7.6 策略淘汰与降级机制

实战层不能只设计准入，也必须设计淘汰。一个策略如果长期不稳定，应及时降级或退出主线，避免系统越跑越复杂。

建议淘汰条件：

- Validation 连续多个时间块不优于基线。
- 多时间块波动过大，命中率或 rank 指标依赖少数极端块。
- 只提升 TopK 命中率，但 logloss、Brier score 或校准明显变差。
- 新增特征后复杂度上升，但稳定性没有同步改善。
- 策略需要频繁人工改参数才能维持表现。
- 数据更新后表现明显回落，且无法用数据质量问题解释。

建议处理方式：

- 降级为观察策略：继续记录，但不作为正式输出依据。
- 回滚到更简单的基线或上一版稳定策略。
- 移出实战层，只保留在研究层做后续对照。
- 在报告中记录淘汰原因、触发指标和对应参数指纹。

### 7.7 日报与实战输出格式

如果进入每日固定预测阶段，日报必须区分“正式策略输出”和“实验策略观察”，避免用户把研究结论误当成实战建议。

每期日报建议包含：

- 彩种和目标期号。
- 使用数据截止期号。
- 数据 canonical 指纹。
- 参数指纹和策略版本。
- 是否为正式策略、观察策略或实验策略。
- 直选 TopN 候选。
- 组选推荐和组选覆盖概率口径。
- 位置池推荐。
- 最近 N 期表现摘要。
- 风险提示：不保证中奖、不保证盈利，仅用于历史研究和固定策略追踪。

推荐输出文件：

```text
reports/
  daily/
    fc3d/
      2026-07-19.md
      2026-07-19.json
    pl3/
      2026-07-19.md
      2026-07-19.json
```

Markdown 用于阅读，JSON 用于后续页面、统计和自动对账。

## 8. 模块建议拆分

### 8.1 数据模块

职责：

- 读取 CSV
- 标准化字段
- 校验规则
- 计算 canonical 指纹

### 8.2 特征模块

职责：

- 构建历史状态
- 生成候选特征
- 控制窗口和衰减
- 保证无泄漏

### 8.3 模型模块

职责：

- 生成候选排序
- 输出候选概率
- 聚合组选概率
- 生成位置池

### 8.4 搜索模块

职责：

- 枚举/采样参数空间
- 计算验证目标
- 记录搜索过程
- 产生可复现实验结果

### 8.5 回测模块

职责：

- 按时间逐期前推
- 记录每期指标
- 汇总 block 统计
- 生成门禁判断

### 8.6 报告模块

职责：

- 生成 Markdown 报告
- 生成 JSON 产物
- 计算 report fingerprint
- 提供归档路径

### 8.7 性能与提速设计

当前 v4 learned ranker 的主要耗时，不是参数搜索本身，而是按时间前推时，反复为 `000-999` 全候选构造特征矩阵。

#### 8.7.1 主要瓶颈

- 每个目标期都要重新读取该期之前的历史。
- 每个目标期都要重新构造候选特征。
- 每个候选都要遍历多个窗口并计算多类统计。
- 研究层还会在 search / validation 上重复做同类计算。

#### 8.7.2 优化优先级

1. 增量历史状态复用：相邻目标期之间只追加新增的中间开奖，不重复全量扫描历史。
2. 候选特征向量化：把 `for candidate in 000-999` 改为 NumPy 批量计算，减少 Python 循环、dict 和临时对象。
3. 按目标函数裁剪计算：例如 `direct_hit_only` 不再计算 group 和 pool 的完整输出。
4. 分时间块并行：search / validation 可以并行，但要控制进程复制带来的内存放大。

#### 8.7.3 设计原则

- 先保证结果一致，再做性能优化。
- 优化只针对 search / validation / walk-forward，不把 Frozen Test 变成调参依据。
- 提速方案必须保持可复现、可指纹化、可回放。
- 优先减少重复计算，其次才考虑并行。

#### 8.7.4 训练提速改造方案

如果要把“每个目标期都重新扫一遍历史”的实现改成更合理的版本，推荐按下面的顺序做：

1. 历史前缀缓存

   - 以目标期为单位缓存历史状态。
   - 缓存键至少包含：彩种、特征配置、目标期号、历史数据指纹。
   - 相邻目标期只追加新进入窗口的开奖，不重复做全量 normalize / sort / tuple 转换。

2. 增量统计状态

   - 位置频率、pair 频率、和值、跨度、形态分布、遗漏状态都改成可滚动更新。
   - 目标期前移时，只更新受新增开奖影响的统计量。
   - 保留一个只读的历史快照，避免不同目标期之间互相污染。

3. 候选特征批量化

   - `000-999` 不再逐个 Python `for` 循环构造 dict。
   - 先把 1000 个候选拆成 NumPy 数字矩阵，再统一计算位置、pair、和值、跨度、形态等特征。
   - 最终只在需要输出结果时再转成 DataFrame。

4. 目标函数裁剪

   - `direct_hit_only` 只算 direct 所需排序，不再强制算 group / pool。
   - `group_hit_only` 只算组选聚合相关输出。
   - `pool_hit_only` 只算位置池质量。
   - `balanced` 和最终正式评估才保留全量输出。

5. 分段并行

   - Search / Validation 可按时间块并行。
   - 并行前先确认缓存粒度，避免每个进程复制整份历史和大矩阵。
   - 并行只作为加速器，不作为正确性前提。

#### 8.7.5 何时该做这类优化

- 当单次全量 search 明显慢到影响迭代节奏时。
- 当特征配置和评估协议已经稳定，不再频繁改结构时。
- 当当前瓶颈已经确认在特征构造，而不是 I/O、指纹、报告生成时。
- 当你想扩大 search 规模，但不希望线性增加等待时间时。

#### 8.7.6 推荐窗口策略

如果当前目标是先把训练速度提上来，同时保留足够的局部规律表达能力，推荐把窗口策略收敛成下面这种形式：

- 全量历史池仍然保留到 7500 期左右，用于 walk-forward 回测和稳定性验证。
- 每个目标期的主训练窗口优先只取前 150 期。
- 窗口内按时间衰减加权，越近的开奖权重越高，越远的权重越低。
- 150 期作为主窗口，不代表它一定是最终最优，只是一个更适合先跑通、先提速、先验证的默认起点。
- 研究层仍然要保留少量对照窗口，例如 100 / 150 / 250 / all，用 walk-forward 比较哪种更稳。

推荐的衰减方式：

- 优先使用平滑的指数衰减，而不是生硬的分段截断。
- 如果需要更强的局部偏置，可以把最近 20 期权重提高，再逐步衰减到 150 期尾部。
- 若 150 期窗口在稳定性上明显不如更长窗口，再把主窗口扩到 250 期，或采用“150 期主窗口 + 长窗口摘要特征”的混合方案。

这个策略的核心不是“只看 150 期”，而是：

- 训练特征优先聚焦最近历史。
- 长历史主要承担稳定性验证和对照作用。
- 窗口长度本身就是需要被验证的研究变量，不是先验固定真理。

#### 8.7.7 150 期窗口落地规则

如果采用 150 期作为第一版主窗口，工程实现必须按下面规则固定口径：

- 对目标期 `T`，特征历史只能取 `T` 之前最近 150 期。
- 特征历史不得包含 `T` 当期开奖，也不得包含 `T` 之后任何开奖。
- 当 `T` 之前历史不足 150 期时，默认跳过该目标期，不进入 Search / Validation / Frozen Test。
- 150 期窗口内使用时间衰减权重，越接近 `T` 的开奖权重越高。
- 默认优先使用指数衰减；初始 `half_life` 可从 30 / 50 / 75 中做 Search / Validation 对照。
- 100 / 150 / 250 / all 只作为研究层窗口对照，不允许用 Frozen Test 结果选择窗口。
- 如果引入长窗口摘要特征，必须明确命名为 long summary，不得和 150 期主窗口混淆。

推荐语义：

```text
target_issue = T
feature_history = latest 150 periods before T
recency_weight = exponential_decay(distance_to_T, half_life)
```

这意味着全量 7500 期不是每个目标期都要直接参与特征计算。全量历史主要用于形成足够多的 walk-forward 目标期，并验证策略在长时间跨度上是否稳定。

#### 8.7.8 性能优化验收标准

性能优化必须同时验收速度和结果一致性，不能只看运行时间。

- 同一份输入数据、同一组参数、同一组目标期，优化前后候选排序应保持一致；如采用浮点向量化导致微小差异，必须记录最大误差并确认 TopK 不变。
- 单目标期特征构造耗时应显著下降，作为第一版目标，可先要求下降 50% 以上。
- 全量 Search / Validation 的总耗时应显著下降，作为第一版目标，可先要求下降 30% 以上。
- smoke 模式只验证流程、指纹和字段完整性，不能用于效果结论。
- 正式模式必须重新跑 Search / Validation，对比优化前后的指标差异。
- 如果速度提升来自裁剪输出，报告中必须标明裁剪模式，例如 `direct_only`、`group_only`、`pool_only`。
- 如果引入并行，必须确认结果顺序、随机种子、日志和产物指纹稳定。

### 8.8 特征数量与权重控制

彩票数据属于弱信号、高噪声问题，特征不是越多越好。特征数量过多时，模型更容易把随机波动当成规律，从而在 Search 上看起来提升，但在 Validation 或长期实战中回落。

#### 8.8.1 特征设计原则

- 优先保留少量稳定、可解释、可复现的核心特征。
- 新增特征必须通过 walk-forward 对照证明有效，不能只因为直觉合理就进入主线。
- 不允许为了追求单次命中率无限堆叠特征。
- 特征复杂度增加后，必须同步观察稳定性、校准和分时间块表现。

#### 8.8.2 推荐特征分组

建议把特征按功能分组，而不是让每个特征完全自由竞争：

- 核心分布类：位置频率、pair 频率、和值分布、跨度分布。
- 趋势类：短长窗口差异、位置趋势、pair 趋势、regime gap。
- 形态类：组三 / 组六 / 豹子转移、近期形态偏移。
- 遗漏类：位置遗漏、冷号修正。
- 上一期关联类：与最新开奖号的距离、重复数字、同位重复。
- 人工约束类：极端和值、豹子、极端跨度等惩罚。

#### 8.8.3 推荐权重占比

权重搜索时建议设置组级约束，避免某类噪声特征权重过高：

- 核心分布类：50% - 70%。
- 趋势类：10% - 25%。
- 形态类：5% - 15%。
- 遗漏类：0% - 10%。
- 上一期关联类：0% - 10%。
- 人工约束类：0% - 10%。

这些占比不是固定真理，而是初始搜索边界。后续应通过 Search / Validation 的稳定性对照调整，不能用 Frozen Test 反推。

#### 8.8.4 第一版推荐特征集

如果先做轻量提速版，建议从 8 到 10 个核心特征开始：

- `position_frequency`
- `pair_frequency`
- `sum_distribution`
- `span_distribution`
- `recent_trend`
- `position_trend`
- `pair_trend`
- `shape_transition`
- `constraint_penalty`

以下特征建议作为弱修正或对照实验，不应在第一版中给过高权重：

- `position_omission`
- `shape_recent_deviation`
- `latest_distance`
- `repeat_latest`
- `trend_30_300`
- `regime_gap_50_all`

如果采用 150 期主窗口，`trend_30_300` 和 `regime_gap_50_all` 应改成更通用的短长窗口特征，或改为 `trend_30_150`、`regime_gap_50_150` 这类与实际窗口一致的特征。

#### 8.8.5 特征准入标准

一个新特征进入主线前，至少应满足：

- 加入后 Search 有改善。
- Validation 也有改善或至少不明显回落。
- 多个时间块表现稳定。
- 不只提升单一 TopK 命中率。
- 不显著恶化 logloss、Brier score 或校准表现。
- 不依赖 Frozen Test 结果做选择。

#### 8.8.6 特征白名单与实验特征

为了避免特征越加越多，建议把特征分成三类管理：

1. 主线特征

   第一版轻量主线建议默认启用：

   - `position_frequency`
   - `pair_frequency`
   - `sum_distribution`
   - `span_distribution`
   - `recent_trend`
   - `position_trend`
   - `pair_trend`
   - `shape_transition`
   - `constraint_penalty`

2. 弱修正特征

   默认允许低权重参与，但不能主导排序：

   - `position_omission`
   - `shape_recent_deviation`

3. 实验特征

   默认关闭，只能在研究层对照实验中启用：

   - `latest_distance`
   - `repeat_latest`
   - `trend_30_300`
   - `regime_gap_50_all`

采用 150 期主窗口后，实验特征中的长窗口项应优先改成通用命名，例如：

- `short_long_trend`
- `recent_regime_gap`
- `long_window_summary`

主线版本中不建议保留语义和实际窗口不一致的特征名，例如窗口已经改成 150 期，但仍输出 `trend_30_300`。

## 9. 指标体系

### 9.1 主指标

建议把以下几个作为主指标：

- `logloss`
- `Brier score`
- `mean rank`
- `direct hit rate`
- `group hit rate`

### 9.2 辅助指标

- `rank percentile`
- `stable blocks`
- `p-value`
- `random baseline gap`
- `probability calibration`

### 9.3 指标解释原则

- TopK 命中不是唯一标准。
- 概率校准比“看起来排序更高”更重要。
- 单次极值不等于稳定优势。

## 10. 时间切分与防泄漏

### 10.1 原则

任何模型都只能看目标期之前的数据。

### 10.2 Search / Validation / Frozen Test

- Search：探索参数空间。
- Validation：只确认 Search 已锁定的最终参数。
- Frozen Test：最终一次性评估。

### 10.3 允许与禁止

#### 允许

- 在 split 边界后追加新开奖。
- 对新数据做新一轮完整实验。

#### 禁止

- 通过测试结果反推参数。
- 反复查看 Frozen Test 来挑模型。
- 把测试段参数回流到训练阶段。

## 11. 版本与指纹设计

### 11.1 必须固定的内容

- 数据版本
- 规则版本
- 特征配置
- 参数配置
- 源码版本
- 评估分段

### 11.2 指纹建议

- `csvSha256`
- `canonicalDataSha256`
- `sourceFingerprint`
- `paramsFingerprint`
- `artifactFingerprint`
- `reportFingerprint`

### 11.3 作用

一旦结果异常，能立刻判断是：

- 数据变了
- 规则变了
- 参数变了
- 代码变了
- 评估段变了

## 12. 推荐实现路线

### 12.1 第一阶段

- 固化现有数据标准化流程。
- 补齐基线层。
- 统一评估协议。

### 12.2 第二阶段

- 建立研究层统一接口。
- 接入 LightGBM 或其他非线性模型。
- 建立多切分 walk-forward 评估。

### 12.3 第三阶段

- 只保留稳定胜出的模型。
- 冻结实战策略。
- 建立长期报告和版本追踪。

## 14. 风险与应对

### 14.1 风险：过拟合

应对：多切分验证、简化模型、看校准和稳定性。

### 14.2 风险：数据泄漏

应对：严格时间切分、冻结测试、指纹校验。

### 14.3 风险：单次结果误导

应对：多窗口、多切分、多指标。

### 14.4 风险：模型复杂化后不可解释

应对：保留基线层和可解释输出。

### 14.5 风险：把噪声当规律

应对：把随机基线和稳定性指标放在前面。

## 15. 结论

对于这个彩票预测系统，更好的方向不是盲目引入更复杂模型，而是建立一个更严格的研究体系：

- 基线层负责判定是否真的有信号。
- 研究层负责比较不同方法的稳定性。
- 实战层只接收长期稳定优于基线的结果。

如果没有模型能在多个时间切分上稳定胜出，那正确结论不是“模型还不够大”，而是“这个问题的可预测信号本来就很弱”。

这套三层设计的价值，就是让这个结论可以被严格验证，而不是靠感觉。

## 16. 当前落地状态（2026-07-19）

当前开发模式不保留旧版本兼容路径，研究层只保留 `learned_ranker_v4`，不接入 LightGBM、序列模型或 ensemble。已落地内容：

- 数据层：抓取只追加 raw JSONL；多个 provider 先生成冲突/来源不足报告，多源对账后才能生成标准 CSV；保留 CSV、canonical、v4源码和三层源码指纹。
- 基线层：固定提供 uniform、shape prior/transition、sum-span、position 20/50/150/all，并统一输出 LogLoss、Brier、mean rank、TopK、校准和 ECE。
- 研究层：目标期只读取此前最近 150 期，窗口固定 20/50/150、half-life=50；相邻目标使用滚动状态，目标矩阵支持指纹化NPZ断点恢复；Search 选参，Validation 只确认。
- 性能层：`000-999` 候选位置、pair、和值、跨度、遗漏和约束使用 NumPy 批量计算；不得通过读取 Frozen Test 提速或调参。
- 实战层：`research/observation/active/demoted/retired` 固定状态机；direct、group、position 分别使用独立随机基线和闸门；日报写入 `reports/daily/<lottery>/`，迁移历史和回滚目标写入策略注册表。
- 安全入口：训练、在线自适应开发、Frozen评估和日报必须显式分开执行，不提供自动串联 Frozen Test 的一键命令。
- 自适应研究：开发区外层stride=1逐期预测，每期更新20/50/150状态；默认每10期只用块起点之前最近500期执行Inner Search/Validation。候选参数包含均匀收缩`λ`，普通训练默认以相对均匀基线的LogLoss、Brier、排名、ECE和时间稳定性组成`research_calibrated`目标。
- 放弃机制：Inner Search/Validation、LogLoss、Brier、ECE、稳定块或`λ>0`任一不满足时，整个预测块退回严格均匀概率，标记`abstained=true`，不得进入主推荐。

三层离线验收只验证结构、数据边界、基线和产物契约，`accepted=true` 不表示预测模型已通过实战准入。只有 Search、Validation、多时间块、统一概率指标和一次性 Frozen Test 全部满足固定闸门，策略才允许进入 `active`。

# 概率算法优化设计方案

> 适用范围：福彩3D、排列三。  
> 当前状态：`probability_v5` 仅为隔离的开发挑战器，不接入日报、影子状态、Frozen Test 或正式推荐。  
> 证据边界：历史开发数据已经被反复研究，只能用于淘汰方案，不能证明新模型有效。

## 1. 结论先行

v5 首版采用以下最小方案：

```text
4个预注册概率专家
  -> 在线自适应指数加权
  -> 独立Calibration只选择temperature
  -> 原始概率指标 + 日常策略Top50双口径评估
  -> 严格统计闸门
  -> 永久保持开发模式
```

四个专家固定为：

1. `UniformExpert`：000-999 均匀概率，永久存在。
2. `EwmaPositionExpert`：三个位置的 EWMA 边际概率乘积。
3. `EwmaPairwiseExpert`：三个位置对的 EWMA 联合概率乘积后归一化。
4. `LegacyGradientExpert`：当前在线梯度机制的冻结结构对照，从头机械更新，不读取 v4 状态。

首版明确不做：

- 不加入 FTRL。既有 FTRL 证据未通过，不能直接包装为新专家。
- 不同时保留 UniformExpert 和第二层动态 `lambda`，避免重复均匀收缩。
- 不搜索大量近似专家、窗口或初始权重，避免专家克隆放大先验权重。
- 不读取、迁移或兼容 `learned_ranker_v4` 状态。
- 不输出研究 Top50，不启用正式推荐。
- 不打开新的独立 Validation，不读取 Frozen Test。
- 不把尚未运行的 5000 次全流程随机模拟写成已通过。

## 2. 对现有问题的修正判断

### 2.1 当前模型不是1000套独立参数

当前 learned ranker 对 000-999 构建候选特征，再用共享特征权重生成 1000 类概率。问题不是“有1000套完全独立参数”，而是：

- 每期仍只有一个真实类别；
- 高维候选空间中的概率质量差异极小；
- 多组相似特征、窗口和约束可能追逐随机波动；
- 小样本 Search/Validation 很难区分真实弱信号与噪声。

因此优化重点应是减少模型选择自由度、使用低阶结构概率和加强验证，而不是仅以“减少1000个参数”为理由改造。

### 2.2 概率指标和Top50必须分开计算，但都参与门槛

LogLoss、Brier 必须使用策略过滤前的完整 1000 类概率。排除上期原号、限制豹子不能修改这组概率，否则 proper scoring rule 失去含义。

同时，最终业务目标是日常策略处理后的 Top50，因此部署判断必须使用同一套候选策略：

```text
raw probability rank
  -> 排除上期完全相同号码
  -> Top50最多保留1个豹子
  -> 从后续排名补足50个
  -> policy-adjusted Top50
```

正确关系是“分开评估、联合否决”：

- LogLoss/Brier 判断概率质量；
- raw Top50 用于定位模型排序变化；
- policy-adjusted Top50 判断真实日常策略结果；
- 任一硬门槛失败都不能晋级。

### 2.3 Validation必须是一次性硬确认

Validation 只能确认已经由 Search 和 Calibration 完全锁定的方案。Validation 失败后不得修改参数再重跑同一段，也不得进入 Frozen。

v5 当前没有可用的独立历史 Validation。历史报告统一标记：

```text
evidenceStatus=exploratory_reused_development
validationOpened=false
promotionPassed=false
researchRankingEnabled=false
recommendationEnabled=false
```

## 3. v5首版算法

### 3.1 完整概率输出

每个专家在目标期开奖前输出：

\[
P_{j,t}(x),\quad x\in\{000,\ldots,999\}
\]

并满足：

\[
P_{j,t}(x)\ge 0,\qquad \sum_xP_{j,t}(x)=1
\]

所有概率只允许由目标期之前的历史生成。

### 3.2 UniformExpert

\[
P_{0,t}(x)=\frac{1}{1000}
\]

Uniform 是真实专家和永久随机基线，不产生固定文本顺序 Top50。开发报告即使存在非均匀混合概率，也不开放研究排名。

### 3.3 EWMA位置边际专家

每个位置和数字维护指数衰减概率：

\[
S_t=\rho S_{t-1}+(1-\rho)I(y_t=d),\qquad \rho=2^{-1/h}
\]

首版固定半衰期 `h=80`，三位组合概率为：

\[
P(a,b,c)=P_1(a)P_2(b)P_3(c)
\]

它用低阶结构共享样本，不建立完整号码专属参数。

### 3.4 EWMA位置对专家

分别维护 `(百,十)`、`(百,个)`、`(十,个)` 的 10x10 EWMA 联合概率，首版固定半衰期 `h=300`：

\[
\tilde P(a,b,c)=P_{12}(a,b)P_{13}(a,c)P_{23}(b,c)
\]

最后对全部 1000 个号码归一化。该专家用于检验低阶位置相关性，不加入完整三位组合记忆。

### 3.5 Legacy Gradient对照专家

该专家复用当前在线梯度的计算机制和已冻结结构，但：

- 不读取 v4 参数或状态文件；
- 从开发历史起点重新初始化；
- 每期先输出概率，再按开奖号机械更新；
- 不代表旧模型已晋级，也不把旧证据继承给 v5。

它的作用是提供同一预序协议下的基准，不是兼容层。

### 3.6 在线自适应指数权重

初始专家权重固定为：

```text
uniform          0.50
ewma_position    0.20
ewma_pairwise    0.15
legacy_gradient  0.15
```

第 `t` 期混合概率：

\[
P_t(x)=\sum_jw_{j,t}P_{j,t}(x)
\]

专家损失：

\[
L_{j,t}=-\log P_{j,t}(y_t)
\]

权重按累计损失进行指数更新，学习率根据累计专家损失范围自适应，并限制最大值为 `0.05`。实现应称为“自适应指数权重”，不宣称是严格 AdaHedge。

该更新属于冻结规则下的机械在线状态更新，允许执行；人工修改专家、半衰期、先验权重、学习率或校准网格则属于重新建模，必须产生新协议。

## 4. 概率校准

首版只校准 temperature：

\[
P_T(x)=\frac{P(x)^{1/T}}{\sum_kP(k)^{1/T}}
\]

固定网格：

```text
0.75 / 1.0 / 1.5 / 2.0 / 3.0
```

Calibration 依次按平均 LogLoss、平均 Brier、较小 temperature 裁决。正 temperature 不改变候选排序，所以 Top50 不参与 temperature 选择，但仍在 Search 和后续评估中独立过闸门。

不再增加第二个动态 `lambda`。UniformExpert 已承担均匀兜底，重复收缩会让“专家权重”和“最终有效模型权重”无法解释。

## 5. 数据分段与职责

### 5.1 当前开发运行

默认配置：

```text
最早150期             仅初始化专家
随后历史              严格逐期预测和机械更新
开发Search 500期      结构筛查和双口径指标，不再调专家结构
Calibration 250期     只选择temperature
开发Evaluation 500期  最终开发区压力测试
最新Frozen 500期      完全排除，不读取
```

专家集合、EWMA半衰期、初始权重和温度网格在运行前已经预注册。因此首版 Search 报告 `selectionPerformed=false`，作用是验证固定结构，而不是在同一 500 期内继续挑参数。

开发 Evaluation 仍属于复用历史，不得命名为 Validation，也没有晋级资格。

### 5.2 未来正式协议

只有开发结构完全冻结且 5000 次全流程随机模拟完成后，才能开始收集新的独立 500 期 Validation：

1. 锁定源码 SHA、配置 SHA、数据截止期和报告协议。
2. 从锁定后的下一期开奖开始，只做机械在线更新。
3. 累积满 500 个未见目标期后一次性打开 Validation。
4. Validation 只确认，不选择参数。
5. 任一门槛失败即关闭该版本，不得用同一 500 期修正后重试。
6. Validation 通过后才有资格讨论是否读取新的 Frozen Test。

“未来”不是唯一条件，“建模决策前锁定且未见”才是独立证据的核心。

## 6. 评估指标与硬门槛

### 6.1 原始概率指标

均匀基线：

\[
LogLoss_U=\log(1000)\approx6.907755
\]

\[
Brier_U=0.999
\]

每期记录：

- 实际号码概率；
- LogLoss、Brier；
- 相对均匀的 Delta LogLoss、Delta Brier；
- 预测分布指纹；
- 四个专家的事前权重和实际号码概率。

500 期汇总要求 LogLoss 和 Brier 改善的 99% 时间块 bootstrap 下界都大于 0。

### 6.2 Top50指标

Search和开发Evaluation逐期同时保存：

- `rawTopK` 和 `rawTopKHit`；
- `policyTopK` 和 `policyTopKHit`。

最终策略门槛只认 `policyTopKHit`，固定要求：

- 至少 500 期；
- 单侧 `p < 0.01`；
- 相对随机基线提升至少 25%；
- 99% Wilson 下界高于随机基线；
- 3/3 非重叠时间块不低于随机基线。

Top50 随机基线为 `50/1000=5%`。候选策略始终补足 50 个号码，因此不能用 raw 排名命中替代策略后命中。

### 6.3 联合否决

开发信号仅在以下三项同时通过时标记为通过：

```text
policy-adjusted Top50严格门槛
AND LogLoss改善99%下界 > 0
AND Brier改善99%下界 > 0
```

即使开发信号通过，`promotionPassed` 仍固定为 `false`，直到全流程随机模拟和新的独立 Validation 都完成。

## 7. 全流程随机模拟

随机模拟必须覆盖完整选择流程，而不是只模拟最终命中数：

```text
随机历史生成
  -> 四专家预序输出和更新
  -> 自适应指数聚合
  -> Calibration选择temperature
  -> 概率与策略Top50联合闸门
```

预注册次数为 5000，至少报告：

- 随机数据通过全部开发门槛的比例；
- 达到真实 Delta LogLoss、Delta Brier 的比例；
- 达到真实 policy Top50 统计量的比例；
- UniformExpert 和复杂专家最终权重分布；
- 全流程经验 p 值。

当前已经实现确定性全流程重放执行器：

- 每次模拟使用由基础种子和试验编号派生的独立种子；
- 串行和多进程使用相同试验编号，结果必须逐试验一致；
- smoke次数必须少于5000，且固定`nullSimulationPassed=false`；
- 正式模式必须恰好执行5000次，并绑定不可覆盖的开发协议和开发报告；
- 正式模式只接受已验证协议对象和已验证开发报告对象，不接受任意SHA或原始JSON；开发报告必须先用锁定数据、源码、配置与协议完整重跑并逐字段一致，不能仅依赖可被重新计算的自哈希；随后继续复核schema、kind、数据、源码、配置、协议身份和派生闸门；
- 正式模式必须提供检查点目录，每个试验按编号独立只写一次并带`trialSha256`，任务通过`as_completed`完成即落盘，重启只计算缺失编号；
- 全部试验完成后按编号写入`trial_set.json`，其中保存有序试验哈希列表及集合哈希；
- null汇总按专家记录最终权重的均值、标准差、25%/50%/75%分位数、最小值和最大值；
- 随机模拟报告独立只写一次，不修改原开发报告。

正式5000次尚未运行，因此开发报告仍必须写：

```text
fullPipelineNullSimulation.status=not_run
fullPipelineNullSimulation.passed=false
```

当前已完成进程并行、小规模串并行一致性、完成顺序检查点、缺失试验续跑和集合哈希测试；尚未完成全长度性能基准及高并发压力测试。完成这些运行保障后才能执行正式5000次；不能用减少次数的smoke结果冒充正式证据。

## 8. 先预测、后更新时序

每个目标期严格执行：

```text
1. 读取目标期之前的在线状态
2. 四个专家各自输出1000类概率
3. 使用上一期结束后的专家权重生成混合概率
4. 保存原始排名和日常策略排名
5. 再读取当期开奖并计算损失
6. 更新Legacy Gradient内部权重
7. 更新EWMA状态
8. 更新下一期使用的专家权重
```

第 5 步之前不得让当前开奖号进入特征、专家权重、温度选择或排名。回归测试必须证明：修改第 `t` 期开奖结果不会改变第 `t` 期分布指纹，但允许改变第 `t+1` 期。

## 9. 报告和状态边界

当前只写开发协议、开发报告和独立随机模拟报告，不写在线模型状态。完整开发运行必须先登记协议，协议锁定玩法、开发数据SHA与截止期、源码、配置、Frozen边界、500/250/500切分和5000次随机模拟计划。协议及全部报告内容相同可重复读取，内容不同禁止覆盖。

开发报告固定包含：

```text
schemaVersion=probability_v5_development_v1
evaluationKind=development_prequential_challenger
evidenceStatus=exploratory_reused_development
frozenRead=false
legacyStateCompatibility=false
validationOpened=false
promotionPassed=false
researchRankingEnabled=false
recommendationEnabled=false
researchTop50=[]
formalRecommendation=null
```

报告记录数据、配置和源码指纹。源码指纹覆盖v5核心、null核心、两条v5 CLI、本次复用的日常候选策略、特征构建和旧梯度计算模块。通用开发入口永远不声明协议已登记；已登记开发入口必须传入只读协议文件路径，并在内部用锁定开发DataFrame完整重算核对。正式null同样只接受只读协议/报告路径，不接受任何内存“已验证”对象。

开发报告额外固定记录：

- `reportSha256`：对不含自身字段的完整规范化报告计算；
- `derivedGates.searchPassed/evaluationPassed/searchAndEvaluationPassed`：开发信号只在Search与Evaluation同时通过时为真；
- 每期`rawDistributionFingerprint`与`calibratedDistributionFingerprint`；
- Calibration选中温度后的逐期概率损失、双指纹和候选审计；
- `lastPredictionExpertWeights`、`postFinalUpdateExpertWeights`与各专家权重分布摘要。

协议、报告和检查点统一采用同目录临时文件写入、文件`fsync`、原子不可覆盖发布和目录`fsync`，避免崩溃时留下可被误认成完整证据的半文件。

不提供旧 schema 转换、v4 状态迁移、缺字段默认值或静默回退。未来若建立 v5 在线状态，应使用全新 schema 和路径，旧版本不匹配时直接拒绝。

## 10. 当前代码结构

首版保持最小实现，不提前拆成多层目录：

```text
src/analysis/digit_probability_v5.py  核心专家、聚合、校准和评估
src/analysis/digit_probability_v5_null.py  全流程均匀随机模拟
scripts/digit_probability_v5.py       仅开发模式CLI
scripts/digit_probability_v5_null.py  随机模拟CLI
tests/test_digit_probability_v5.py    泄漏、口径、Frozen隔离回归
tests/test_digit_probability_v5_null.py  串并行等价和正式次数约束
```

开发 smoke：

```bash
.venv/Scripts/python.exe -X utf8 scripts/digit_probability_v5.py \
  --lottery fc3d \
  --csv data/fc3d/official_history.csv \
  --output reports/development/probability_v5_fc3d_smoke.json \
  --frozen-test-periods 500 \
  --smoke
```

`--smoke` 只使用 Frozen 之前 50 期验证代码链，不能作为算法证据。默认完整开发运行虽然仍不读取 Frozen，但会执行历史开发评估；在正式实验登记前不应随意反复运行并据此改参数。

完整开发运行必须先只写一次登记协议：

```bash
make digit-probability-v5-register DIGIT_LOTTERY=fc3d
make digit-probability-v5-development DIGIT_LOTTERY=fc3d
```

随机模拟执行链smoke：

```bash
make digit-probability-v5-null-smoke \
  DIGIT_LOTTERY=fc3d \
  DIGIT_V5_NULL_WORKERS=2
```

仓库暂不提供一键正式5000次Make目标，避免在没有性能基准和续跑方案时误启动高成本任务。

## 11. 后续算法调整顺序

只有首版固定结构完成压力测试后，才按以下顺序考虑调整；每次只建立一个新版本：

1. 为已实现的全流程并行随机模拟增加全长度性能基准并锁定合理worker数量。
2. 增加 confidence sequence 或 e-value，解决未来持续观察时的可选停止问题。
3. 检查专家权重集中度和相似度，防止多个相关专家共同稀释 UniformExpert。
4. 若低阶位置对长期有证据，再评估条件分解专家，不直接加入完整号码记忆。
5. FTRL 只有在独立、预注册的开发协议中优于当前四专家基线后，才可进入下一版本。

不建议继续做：

- 在同一历史上扩大 temperature、半衰期或专家网格；
- 根据单个彩种、单个500期块的漂亮结果改结构；
- 把平均排名、MRR 或 ECE 与 LogLoss 直接加权成无量纲总分；
- Validation 失败后更换规则并重复查看；
- 将开发区非空排名直接接入日报。

## 12. 完成与激活条件

v5 从“开发挑战器”进入“可收集独立 Validation”的最低条件：

- 核心、CLI、泄漏和双口径测试通过；
- 默认 500/250/500 开发协议完整运行并保留不可覆盖报告；
- 5000 次全流程随机模拟完成且经验假阳性率满足预注册阈值；
- 源码、配置、数据截止期和未来更新规则全部锁定；
- 不读取任何 v4 状态。

研究排名开放还必须等待新的独立 500 期 Validation 一次性通过。正式推荐属于更高层决策，首版没有实现入口，也不因开发或 Validation 通过而自动开启。

该设计的目标不是保证命中率提高，而是让“没有稳定信号”能够被系统正确表达为均匀基线、空排名和拒绝晋级。
